# Browser-Like Drag Selection Reference

**Date:** 2026-03-10

**Purpose:** 记录浏览器在“拖拽选择文本”场景下的典型交互表现，并明确 Lumina 编辑器希望达到的效果，以及下一步应重点调查的代码文件。

## 1. 浏览器里的典型效果

用户在浏览器中选择一段普通文本时，常见行为如下：

### 1.1 拖拽选择的基本行为

- 用户按下鼠标左键，从某个文本位置开始拖动
- 只要鼠标左键没有松开，浏览器会持续更新选区范围
- 选区 anchor 基本固定，head 会随着鼠标位置实时变化
- 用户感知上是“我拖到哪里，就选到哪里”

### 1.2 拖到可视区域边缘时的行为

当用户继续拖动，并把鼠标带到可滚动文本区域的顶部或底部附近时：

- 浏览器不会让选区停在当前可视区域里
- 内容区域会自动持续滚动
- 滚动发生时，选区会继续扩展
- 用户不需要松手
- 用户也不需要再额外滚动滚轮

这就是用户对“自然选择文本”的默认预期。

### 1.3 速度不是固定值，而是有梯度

浏览器通常不是恒定速度滚动，而是具有明显的边缘速度层次：

- 鼠标刚靠近边缘时，滚动较慢
- 更贴近边缘时，滚动更快
- 鼠标越过边缘更远时，速度会进一步加快

用户感受到的是一种连续的、顺手的加速，而不是机械的固定速度。

### 1.4 选区扩展与滚动是同步的

浏览器里的体验不是“先滚动，再重新修正选区”，而是：

- 内容在滚
- 选区 head 同时继续向拖拽方向扩展
- 两者几乎同步发生

因此用户看到的是一个持续的、连贯的选择过程，而不是滚动和选区互相打架。

### 1.5 停止条件

浏览器里的边缘自动滚动通常会在以下条件下停止：

- 鼠标左键松开
- 鼠标回到不再触发边缘滚动的安全区域
- 已滚到文档顶部或底部，无法继续滚动

## 2. 我们期望的效果

Lumina 编辑器在“拖拽选择文本”这件事上，目标不是做一种“像浏览器但有点区别”的交互，而是尽量对齐浏览器的默认体验。

### 2.1 目标一句话

**用户按住鼠标左键拖拽选区时，只要鼠标继续压着并接近或越过编辑器内容区的上下边缘，编辑器内容区就应该自然地持续滚动，选区也应继续随滚动稳定扩展，速度还应具有明显的边缘梯度。**

### 2.2 具体要求

- 滚动对象必须是编辑器内部可滚动内容区
- 不是整个应用窗口
- 不是外层页面容器误滚
- 不是只有滚轮才能触发

### 2.3 用户感知上的“自然”

如果实现正确，用户应当感受到：

- 拖到边缘时，不用松手，内容会自己继续滚
- 选区会继续变长
- 越靠近边缘，滚动越明显
- 越越过边缘，滚得越快
- 鼠标回来后，速度减弱或停止
- 松手后立即停止

### 2.4 不符合预期的表现

以下都说明实现还不对：

- 鼠标按住拖到边缘，但完全不滚
- 只有滚轮能滚，拖拽本身不滚
- 需要拖出很远才突然开始滚
- 速度恒定，没有梯度
- 滚动对象错了
- 滚动过程中选区会跳、抖、回闪
- 选区没有继续稳定扩展

## 3. 与当前问题的关系

当前的问题不应先从某条内部技术路径出发理解，而应先从“用户看到的交互是否像浏览器”来定义。

也就是说，下一步排查应优先回答：

- 当前真实运行时里，拖拽选择到底由谁接管？
- 在“鼠标压住 + 拖拽中 + 接近边缘”这个状态下，谁负责驱动滚动？
- 滚动和选区更新之间是不是同一条连续的事件链？
- 真实桌面端与测试环境的事件行为是否一致？

## 5. 下一次排查建议

下一次真正做实现或排查时，建议按照这个顺序：

1. 先在真实桌面端确认“按住拖拽到边缘时为什么不滚”
2. 确认当前拖拽时的滚动责任链是否已经被某条手动同步路径截断
3. 明确滚动容器到底是哪一个 DOM 元素
4. 明确边缘触发区和速度曲线应挂在哪条事件链上
5. 先做最小可复现，再加回归用例，再做最小实现
6. 最终仍以真实桌面端拖拽体验验收，而不是只看浏览器内测试通过

## 6. 一句话总结

我们期待的不是“某种自定义拖拽效果”，而是：

**编辑器里的文本拖拽选择，应该尽量像浏览器选择普通网页文本一样——按住、拖到边缘、自然滚动、速度有梯度、选区持续稳定扩展。**
